New York City publishes more open data than almost any government on earth. It’s a genuine commitment to transparency. And for most of the people whose lives that data describes — tenants, parents, neighbors — it’s completely unreachable.

The HPD violation database requires a Socrata API query to use meaningfully. Eviction records are a raw CSV with codes the public has never seen. School-budget data lives across four separate spreadsheets keyed to district identifiers no parent has ever encountered. The 311 complaint archive is 35 million rows.

The problem isn’t secrecy. The data is there, it’s public, it’s updated regularly. The problem is the unglamorous middle mile: turning machine-readable government data into something a tenant can use on their phone in a hallway.

That’s what CivicByte does. We build that middle mile — free, no accounts, no tracking, no jargon — and we review every tool that joins the platform to make sure it holds to those standards.

Every CivicByte tool is built under one constraint: no backend, no accounts, no cost to run at scale. That constraint isn’t a limitation — it’s the point.

Data pipelines run on GitHub Actions, pulling from NYC Open Data on automated schedules. The Building Report indexes 3.2 million property records nightly. Landlord Watch aggregates violation counts across every registered owner in HPD’s database — roughly 80,000 entities. The 311 Lookup processes a 35-million-row dataset into a searchable flat file served from CDN at zero marginal cost.

The frontend is deliberately minimal: HTML, CSS, and vanilla JavaScript. No framework dependencies, no build pipeline needed to read the source. Any student, journalist, or curious neighbor should be able to open the code, understand it, and fork it.

Tools enter the platform through an open submission process, reviewed against four editorial standards: free to use, open source, usable without a degree, actively maintained. Tools that go stale are removed.

CivicByte is a platform, not a finished product. It’s a system for turning civic data into civic access — and it’s only as good as the tools it lists.

The next tools in review include a campaign finance lookup (matching donor addresses to candidates and PACs), a zoning board decision archive (searchable by block and lot), and a city contractor performance database (completion rates, cost overruns, violation history).

The longer mission is about a standard: that any public dataset maintained by any NYC agency should have a human-readable interface within six months of publication. We’re working toward that benchmark one tool at a time.

The submission process is open. The standards are published. The data is already public — we just need to make it usable.

We don’t have user accounts and we don’t want them. That means we measure carefully: aggregate, non-identifying request counts; referrer patterns when a tool gets shared; and direct outreach from people who tell us how they used the data.

Numbers on this page are rounded to the nearest hundred. Any case, hearing, or filing cited here has been verified with the person involved before publication. Testimonials are first-person and unsolicited — we don’t solicit reviews or endorsements.

The point of careful measurement isn’t to pad numbers. It’s to be honest: we’d rather undercount real impact than overstate it. The work speaks for itself or it doesn’t.
